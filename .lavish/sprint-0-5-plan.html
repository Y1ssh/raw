The graph — what we're building

Nodes are deterministic code; the LLM lives only inside agent. Every edge routing decision is structural (in should_continue), not prompt-based — principle #6. The whole graph is compiled with a PostgresSaver checkpointer so every super-step is persisted.

1 · core/state.py

typed AgentState + reducers

Pydantic BaseModel state. Each field annotated with a reducer so concurrent writes in one super-step merge deterministically instead of clobbering. Nodes emit deltas (e.g. {"iteration": 1}) and operator.add sums them — the canonical LangGraph pattern for concurrency-safe counters.

FieldTypeReducerWhy (concurrent-update semantics)
messageslist[dict]append_messagescustom dict-append reducer — appends & de-dupes by id, keeps list[dict] (not LangChain objects) so dict-based tests pass
iterationintoperator.addnodes emit +1 deltas → summed, never lost on a race
planstr | Nonetake_lastlast non-None write wins (planner overwrites, no error on concurrent set)
tool_resultslist[dict]operator.addconcurrent tool executions concatenate — none dropped
error_reasonstr | Nonetake_lasterror_handler sets it; take_last avoids "concurrent write, no reducer" crash
retry_countintoperator.adddelta-based, same as iteration — bounded retry accounting

2 · core/graph.py

nodes · edges · checkpointer

build_graph(call_model) takes the model fn as a slot (reusing ModelFn from loop.py) so the base stays domain-clean and tests inject a fake. The tools node is a documented no-op placeholder — packs replace it with a real MCP-backed ToolNode.

3 · api/routes.py

POST /api/chat

Runs the graph for a thread_id and returns the final state. Graph + checkpointer are built once in the FastAPI lifespan and stored on app.state (mirrors the reference's create_graph() singleton). Sync PostgresSaver is driven off the event loop via run_in_threadpool.
